# Personal M365 Excel Copilot Browser Proxy

这是一个依赖真实浏览器会话的 **个人版 Excel Copilot 本地代理**。它通过 CDP (Chrome DevTools Protocol) 附着到本地 Microsoft Edge 浏览器，把浏览器中的 Copilot 回答包装成兼容 OpenAI 和 基础 Anthropic Messages 的 API 接口。

主要目标是提供一个高性能、低延迟的本地文本对话接口，方便在各类程序或插件中接入 Excel Copilot 的模型能力。

---

## 🚀 核心特性

- **接口兼容**
  - 提供 `GET /health` 与 `GET /v1/models`。
  - 兼容 OpenAI 风格的 `POST /v1/chat/completions` 请求（仅支持流式 `stream=true`）。
  - 兼容 Anthropic 风格的 `POST /v1/messages` 请求（仅支持流式 `stream=true`）。
- **真实浏览器流式提取**
  - 基于持续的 DOM 轮询，支持将页面产生的新信息拆分为分段增量的 SSE 文本流输出。
  - 拥有智能抓取策略，可有效避开搜索过程中的中间态扰或者提示信息，优先精确提取最终回答内容（常为 Markdown 格式）。
- **API 兼容（文本对话）**
  - 支持广泛的多重系统与程序发出的字符编码请求 (包括 UTF-8, UTF-8-SIG, GB18030, GBK 等)。
  - 虽然提供协议层面的 Anthropic `tool_use` 格式兼容，但本质上是**基于文本提示注入的实验性解析**。
  - 建议主要将其作为**文本对话模型**使用。由于浏览器侧 Copilot 的限制，它无法实现原生、高可靠的提示词绕过或物理级的工具执行（端到端工具链）。
- **配置与环境要求极简**
  - 系统侧完全零成本——无需再次独立配置浏览器或者在 Docker 内设置渲染环境，你的系统如果正常安装了微软 Edge 就可以复用。
  - 直接复用当前浏览器的 M365 原生登录状态和文件页面上下文。不需要去抓取 Cookie 和配置各种 Auth。

## 🛠️ 安装与依赖

### 1. 确认系统与业务前置条件
在执行此系统代理之前，需确保你的机器和软件状态已完整：
- 系统环境为 Windows。
- **Edge 浏览器** 已安装并能够正常运行。
- 你已经成功登录了你的对应 M365 个人账号/企业账号。
- 你已经在浏览器中成功新建或打开了一份 **Excel Online** 表格，并且必须将右侧的 **Copilot taskpane (副驾驶助手侧边栏)** 单击处于**已展开**状态。

### 2. 准备 Python 依赖库
另外，需要你的本机上准备有 Python 3.12 左右（或更高版本）环境。请你在终端/命令行中，使用系统 pip （或项目的虚拟环境）执行如下命令安装所需的框架支持：
```bash
pip install fastapi uvicorn websocket-client
```

## 🏃 启动与使用指南

### ▶️ 方式 A：一键式快速启动（推荐）
如果你还没有启动 Edge（并期望脚本能为你初始化一台带调试权限自动连结的 Edge），你可以执行此方法。
在包含被克隆的此脚本对应文件夹根目录下打开具有相关权限的 **PowerShell** 窗口，运行：
```powershell
powershell -ExecutionPolicy Bypass -File .\start_personal_browser_proxy.ps1
```
该命令会自动完成一条龙服务：
1. 发起启动带有开放了远程重定向调试端口 (CDP=9222) 参数的 Edge 浏览器。
2. 调用 Python 核对和检查 Python 必须运行的库环境依赖状态。
3. 如果满足所有条件并监测就绪后，它将启动代理 API 本地代理服务器（其默认进程将会运行在并侦听 `http://127.0.0.1:12803` 端口上）。

> **⏭️ 跳过浏览器拉起过程 (`-SkipBrowser`)**：
> 如果你觉得脚本代启动麻烦，已提前开启配置过参数且拥有调试端口 `:9222` 的 Edge 环境，并且在那里面已将 Excel Online 和对应右侧的 Copilot 侧边栏开启待命了。
> 你可通过添加启动参数直接单步驱动该底层：
> ```powershell
> powershell -ExecutionPolicy Bypass -File .\start_personal_browser_proxy.ps1 -SkipBrowser
> ```

### ▶️ 方式 B：完全自主的手工分步启动
适合希望能更精细深入的调试人员用来排错：

**第 1 步：人工拉起支持 CDP 端口的 Edge：**
这可以通过我们提供的快速短指令：
```powershell
powershell -ExecutionPolicy Bypass -File .\start_edge_cdp_and_check.ps1
```
你必须去核实 `http://127.0.0.1:9222/json` 地址的确可以在浏览器中获取到有回显文本返回（出现一串 json 配置字样）。

**第 2 步：手工准备所需页面上下文：**
*需要你手动在此开起的 Edge 环境中完成对应的登录操作，打开需要操作互动的 Excel Online 页面文件并等待载入，待页面功能加载完成后展开位于右侧的 Copilot 面板。*
*(如果你之前打算以 `-SkipBrowser` 的方式带起系统进程本身也是需要严格先确认此操作的。)*

**第 3 步：跑起中转的代理核心：**
当上述 2 步操作环境准备好的情况下即可让代理跑起来进行捕获了。
```bash
python browser_attached_proxy.py
```

## 🔌 API 接口与测试说明

服务拉起之后将会持续运行和响应在 `http://127.0.0.1:12803` 上。

### 对内部基础监控项的探测检查
你可以跑以下的 curl 口令校验，来检查下服务此时到底是死是活或者是端口被强开的阻塞了：
```bash
# 探测底层组件的连接和运行健康状态（Status 为 ok 代表着其 CDP 的通信和前端浏览器就位的健康状况是符合请求条件的）
curl http://127.0.0.1:12803/health

# 获取和查看已定义挂载暴露的模型支持清单
curl http://127.0.0.1:12803/v1/models
```

### 例：进行一场兼容于 OpenAI 规范流聊天的通信测试
```bash
curl -N -X POST "http://127.0.0.1:12803/v1/chat/completions" \
  -H "Content-Type: application/json; charset=utf-8" \
  -d '{
    "model": "excel-copilot-browser-attached",
    "stream": true,
    "messages": [
      {"role": "user", "content": "请只回答两个字：苹果"}
    ]
  }'
```

### 例：进行一场兼容于 Anthropic 规范及其 Tool 功能参数流的消息模拟测试
*(以下请确保该请求头部包含了规范下的相关 api-key 等定义要求，以触发正常的 `data` 段 SSE 流)*
```bash
curl -N -X POST "http://127.0.0.1:12803/v1/messages" \
  -H "Content-Type: application/json; charset=utf-8" \
  -H "x-api-key: local" \
  -H "anthropic-version: 2023-06-01" \
  -d '{
    "model": "claude-sonnet-4-6",
    "stream": true,
    "max_tokens": 1024,
    "messages": [
      {"role": "user", "content": [{"type": "text", "text": "请介绍一下 Excel Copilot"}]}
    ]
  }'
```

## 🗂️ 根包目录架构与结构讲解
此源码框架为保障自身高拓展性和为应对将来极有可能出现的微调工作，而存放了很多单独拆解出的排障分析测试辅助，你可以依据如下进行有针对理解操作的查找。
- `browser_attached_proxy.py`: （主要）本程序的 FastAPI 本体，也是所有从外部过来的流派接收控制核心处理器点。所有的解码规范判断兼容转化封装处理都在这里，并处理向下交棒给底部的控制逻辑调运工作。
- `browser_driver.py`: （重点）实际下行真正操作的核心驱动部分文件，全权依靠并去接驳底层向外的那个 `:9222` 重定向的通信控制句柄 WebSocket 去处理发包连接任务，并担负在成片 DOM 代码内容中去抓取对应的回答文字内容的筛选数据分片并将其组合返流的工作。
- `debug/`: （开发调试分析包）用于放大量的排查特定卡点原因的代码工具脚本，其中包括在整个架构最上层的一道模拟发送业务的全路径流程检测器： `run_live_proxy_regression.py` （常被用于作为该套组件在修改了新参数后执行的本地线上运行总合验收测试程序，你可以通过它实时跑一次自动向 Taskpane 去写入请求的过程动作，观察和打印出来网络上报的情况信息反馈等等状态是否合格；还包含对诸如各种按键控制等独立控件功能的调试查验动作类文件分析包等）。
- `tests/`: 代理 API 与模块部分底层的独立纯本地逻辑模拟运行验证部分的单元测试（与是否带起并能够通过 CDP 没有直联牵扯，验证本身代码架构转换功能使用）。

## ⚠️ 常见排错答疑 (FAQ & Troubleshooting)

| 问题症状/情况描述 | 原因分析与排查建议 |
|---|---|
| 服务拉起健康监测时： **`/health` 访问抛出了异常错误 或 直接就是返回 HTTP `503` 代码** | 1. 优先判断并排查，请确认：你当前电脑上的操作系统此时是不是真的存在且启动起带带且开启了 CDP 端口机制功能支持参数的那套 Edge 环境进程了。<br>2. 访问页面探测 `http://127.0.0.1:9222/json` 地址是否有可观反应。<br>3. 最容易遗漏点的点（也是最坑的点）是，一定要判断页面不仅正常载入完成了，而且**必须是已经登录进去展开出来右边那一条 Copilot 侧边信息操作控制操作栏了！**，否侧无法顺利解析 `iframe` 就会抛该类连击不通的问题。 |
| 对接口发送了带有 prompt 输入后服务陷入**长久等待阻塞，最终返回没抓取到或者为空结果的内容。** | 在排除上面大步骤没落空的情况下，问题极有可能是 `browser_driver.py` 在通过选择提取绑定到某个正在发生文字刷新重写的地方跟目前这个轮询生成内容发生了节点脱节绑定造成的异常：你当前页面确实能接收请求了但页面反应很慢或者压根此时在微软那边的服务抽风卡住了并没有实际上真正生成字出来；（也有一定极小情况概率你此刻使用的连接向进程属于上一轮的残留没完全清空的残余服务端，可以把全部关闭了重启整个一次试试）。<br>建议跑该专用排查脚本查验证 `python debug/debug_taskpane_wait.py` 借它来检测网络在投递信息之后真实有没有产生任何后续的回应加载信息过程； |
| （使用 `-SkipBrowser` 的时候）抛错** `powershell` 执行启动崩溃并提示闪退和错误报告。** | 如果你遇到该状况基本问题就是，当前你已经有一个占用着该预选端口提供代理服务的程序实例在工作跑着了，无需再强开会引起底层冲突，关掉重复端口占用并可以根据自身需求去进行直接请求尝试。 |
| 当你在提交含有涉及到搜索引擎或者大量抓取的问答诉求类**结果老是被腰斩或中途被截断定格在了诸如这句 “我正在搜索...** 类描述当中时 | 这是受当前此系统开发版本的特性机制影响导致的，我们在现行的版本的处理规则判断是去尝试捕获页面上出现的 `isGenerating` 阶段生成字样的情况并将其排除作为最终正文答案，不过受制微软官方偶尔有对加载样式做出短变频调整时会造成此处偶尔失灵提前给截断判断掉；如果需要稳定，目前只能**放慢涉及搜索型和多次需要搜索查验类型交互问句的频率深度**，或者等待此部分判定算法后续跟做针对调整优化。 |
| 第三方工具调用本端点模型去跑时，如： Cherry Studio 使用途中老报错 **服务器抛出 `500` 未知错误处理或者内容字符被转录失败。** | 本项目的当前发行版本是已经做了重大规模完善性的重构的，完全兼容应对现在繁杂的多重中外部流通信所需要的所有系统常见字符包环境 `utf-8`,`utf-8-sig`,`gb18030`,`gbk` 的正确编解码响应适配映射关系机制了；假如你依然还在这里发生大量乱码崩溃的情况建议第一时间审视反思一下，**你确认此时所正在调用着的这个代理进程，是来自于该套包下刚刚通过执行这里指令重新被启动起的全新这个主进程程序包体文件**而不是误带挂起在了旧版本的那个残留节点进程下么。 |
